# 2주차 핵심 키워드 정리

## 1. SQL 쿼리

내가 이해한 SQL은 데이터베이스에게 “어떤 데이터를, 어떤 조건으로, 어떤 형태로 달라”고 요청하는 언어이다.  
백엔드에서는 결국 화면에 필요한 데이터를 가져오는 일이 많기 때문에 SQL을 안다는 건 단순히 문법을 외우는 게 아니라, 서비스 요구사항을 데이터 조회 방식으로 바꿀 수 있다는 뜻이라고 생각했다.

### 1) CRUD

CRUD는 데이터베이스의 가장 기본적인 작업 4가지를 의미한다.

- Create: 생성
- Read: 조회
- Update: 수정
- Delete: 삭제

### Create

```sql
-- users 테이블에 새로운 회원 1명을 추가하는 쿼리
INSERT INTO users (
    provider,       -- 로그인 제공자 예: kakao
    provider_id,    -- 소셜 로그인에서 내려주는 고유 사용자 번호
    name,           -- 실제 이름
    nickname,       -- 서비스에서 사용할 닉네임
    phone,          -- 전화번호
    gender,         -- 성별
    status,         -- 회원 상태
    point,          -- 현재 보유 포인트
    created_at,     -- 생성 시각
    updated_at      -- 수정 시각
)
VALUES (
    'kakao',
    '123456',
    '김찬혁',
    '찬찬',
    '010-1111-2222',
    'MALE',
    'ACTIVE',
    0,
    NOW(),          -- 현재 시각 저장
    NOW()           -- 현재 시각 저장
);
```
회원가입처럼 새로운 데이터를 추가할 때 사용하는 쿼리이다.

### Read
```sql
-- users 테이블에서 id가 1번인 사용자의 정보만 조회
SELECT
    id,             -- 회원 ID
    nickname,       -- 닉네임
    point           -- 현재 포인트
FROM users
WHERE id = 1;       -- id가 1인 사용자만 가져오기
```
특정 사용자의 기본 정보를 조회할 때 사용한다.

### Update
```sql
-- users 테이블에서 id가 1번인 사용자의 닉네임을 수정
UPDATE users
SET
    nickname = '새닉네임',   -- 닉네임 변경
    updated_at = NOW()      -- 수정 시각도 현재 시각으로 갱신
WHERE id = 1;               -- id가 1인 사용자만 수정
```
기존 데이터를 수정할 때 사용한다.

### Delete
```sql
-- users 테이블에서 id가 1번인 사용자를 실제로 삭제
DELETE FROM users
WHERE id = 1;   -- id가 1인 사용자만 삭제
```
1번 사용자를 데이터베이스에서 완전히 지운다.

```sql
-- 사용자를 실제로 지우지 않고 탈퇴 상태로만 바꾸는 쿼리
UPDATE users
SET
    status = 'INACTIVE',   -- 회원 상태를 비활성화로 변경
    deleted_at = NOW()     -- 탈퇴 시각 기록
WHERE id = 1;              -- id가 1인 사용자만 처리
```
데이터를 삭제할 때 사용하는 쿼리다.
다만 실무에서는 실제 삭제보다 deleted_at을 두고 소프트 삭제를 더 자주 사용한다고 생각한다.

### 2) CRUD

필터링은 WHERE절을 사용해서 원하는 데이터만 골라오는 것이다.
```sql
-- 보상 포인트가 500 이상인 미션만 조회
SELECT *
FROM missions
WHERE reward_point >= 500;   -- reward_point가 500 이상인 행만 선택
```
포인트 보상이 큰 미션만 골라서 가져오는 쿼리다.

### 3) Sorting
정렬은 ORDER BY를 사용한다.
```sql
-- missions 테이블의 미션들을 최신 생성 순으로 정렬
SELECT *
FROM missions
ORDER BY created_at DESC;   -- created_at 기준 내림차순 정렬
```
가장 최근에 만들어진 미션부터 보여주는 쿼리다.
- ```ASC```는 오래된 순, 작은 값부터
- ```DESC```는 최신 순, 큰 값부터

### 4) Limit
한 번에 너무 많은 데이터를 가져오지 않기 위해 LIMIT을 사용한다.
```sql
-- 미션을 최신순으로 정렬한 뒤, 앞에서부터 10개만 조회
SELECT *
FROM missions
ORDER BY created_at DESC
LIMIT 10;   -- 결과를 최대 10개까지만 가져오기
```
최신 미션 10개만 잘라서 보여주는 쿼리다.

```sql
-- 미션을 최신순으로 정렬한 뒤,
-- 앞의 20개를 건너뛰고 그 다음 10개를 가져오는 쿼리
SELECT *
FROM missions
ORDER BY created_at DESC
LIMIT 10 OFFSET 20;
```
한 페이지당 10페이지라면
이 쿼리는 21~30까지 가져오고 3페이지에 속함

- LIMIT 10 → 10개 가져와라
- OFFSET 20 → 앞의 20개는 건너뛰어라

1페이지라면 ```LIMIT 10 OFFSET 0```\
2페이지라면 ```LIMIT 10 OFFSET 10```\
3페이지라면 ```LIMIT 10 OFFSET 20```

### 5) Aggregate
집계 함수는 여러 행을 모아서 개수나 합계를 구할 때 사용한다.
```sql
-- user_missions 테이블에서 완료한 미션만 모아서
-- 사용자별로 몇 개를 완료했는지 집계
SELECT
    user_id,                    -- 어떤 사용자 기준인지
    COUNT(*) AS completed_count -- 완료한 미션 개수 세기
FROM user_missions
WHERE status = 'COMPLETED'      -- 완료 상태인 데이터만 대상으로 함
GROUP BY user_id;               -- user_id별로 묶어서 계산
```
각 사용자가 완료한 미션 개수를 세는 쿼리다.
- ```COUNT(*)```는 행 개수를 세는 함수
- ```GROUP BY user_id```는 같은 user_id끼리 묶는 것\
예를 들어 user_id가 1인 행이 완료 상태로 3개 있으면\
결과는 ```1 | 3``` 이런 식으로 나온다.

### SQL의 장점
- 원하는 데이터를 조건에 맞게 정확하게 가져올 수 있다
- 정렬, 집계, 필터링을 한 번에 처리할 수 있다
- 관계형 데이터베이스의 강점을 가장 직접적으로 사용할 수 있다

### SQL의 단점
- 쿼리가 복잡해질수록 가독성이 떨어질 수 있다
- join이 많아지면 구조를 잘 이해해야 한다
- 화면 요구사항이 복잡해질수록 성능까지 같이 고민해야 한다

### 느낀점
이번 주차를 하면서 느낀 건, SQL은 단순히 SELECT *로 끝나는 게 아니라는 점이었다.
같은 기능을 구현해도 어떤 테이블을 기준으로 볼지, 어떤 조건으로 걸러낼지, 어떤 순서로 보여줄지에 따라 쿼리가 완전히 달라진다.
그래서 SQL은 문법 문제라기보다 설계와 해석의 문제에 더 가깝다고 느꼈다.

## 2. 트랜잭션
트랜잭션은 여러 SQL 작업을 하나의 논리적인 작업 단위처럼 묶어서 처리하는 개념이다.
쉽게 말하면 “이 작업들은 같이 성공해야 하고, 하나라도 실패하면 전부 되돌려야 한다”는 보장을 주는 장치라고 이해했다.

## 왜 필요한가?
예를 들어 사용자가 미션을 완료했을 때 아래 두 작업이 같이 일어난다고 가정하면,

1. user_missions의 상태를 COMPLETED로 변경
2. users.point에 보상 포인트를 추가

이 둘 중 하나만 반영되면 데이터가 꼬인다.

- 미션은 완료됐는데 포인트는 안 올라간다
- 포인트는 올라갔는데 미션 상태는 그대로다

이런 문제를 막기 위해 트랜잭션이 필요하다.

```sql
-- 트랜잭션 시작
START TRANSACTION;

-- 1단계: 특정 사용자의 미션 상태를 완료로 변경
UPDATE user_missions
SET
    status = 'COMPLETED',     -- 진행 상태를 완료로 바꿈
    completed_at = NOW(),     -- 완료 시각 기록
    updated_at = NOW()        -- 수정 시각 갱신
WHERE id = 10                 -- user_missions의 10번 데이터
  AND user_id = 1;            -- 그리고 그 데이터의 사용자도 1번인지 확인

-- 2단계: 해당 사용자에게 포인트 지급
UPDATE users
SET
    point = point + 500,      -- 기존 point 값에 500 추가
    updated_at = NOW()        -- 수정 시각 갱신
WHERE id = 1;                 -- 1번 사용자에게 적용

-- 위 두 작업이 모두 성공하면 최종 반영
COMMIT;
```
미션 완료 처리와 포인트 지급을 하나의 작업처럼 묶어서 실행하는 쿼리다.

```sql
-- 트랜잭션 도중 문제가 생기면 지금까지 한 작업을 전부 취소
ROLLBACK;
```
중간에 오류가 나면 아예 처음 상태로 되돌린다.

### ACID
트랜잭션은 보통 ACID라는 네 가지 성질로 설명한다.

- Atomicity 원자성

    전부 성공하거나 전부 실패해야 한다.
- Consistency 일관성

    작업 전후에 데이터 규칙이 깨지면 안 된다.
- Isolation 격리성

    동시에 실행되는 작업끼리 서로 지나치게 간섭하면 안 된다.
- Durability 지속성 

    성공한 결과는 이후에도 유지되어야 한다.

### 트랜잭션의 장점
- 데이터가 어중간하게 저장되는 상황을 막아준다
- 여러 작업을 하나의 기능처럼 안전하게 처리할 수 있다
- 서비스 상태를 일관되게 유지할 수 있다


### 트랜잭션의 단점
- 동시에 많은 작업이 일어나면 락 문제나 성능 저하가 생길 수 있다
- 너무 길게 잡으면 다른 쿼리 실행에 영향을 줄 수 있다
- 무조건 많이 쓰기보다 꼭 같이 처리해야 하는 작업에 적절히 써야 한다

### 느낀 점
트랜잭션은 단순히 “쿼리를 여러 개 묶는 기능”이 아니라, 서비스 상태가 어중간하게 남지 않도록 보호하는 안전장치라고 느꼈다.
특히 백엔드에서는 사용자 입장에서 하나의 기능처럼 보이는 작업이 실제로는 여러 SQL로 나뉘어 있기 때문에, 트랜잭션 개념이 꼭 필요하다고 생각했다.

## 3. 인덱스
인덱스는 원하는 데이터를 더 빨리 찾기 위한 색인이다.
책 뒤에 있는 색인 페이지처럼, 데이터베이스도 인덱스를 통해 특정 조건에 맞는 데이터를 더 빠르게 찾을 수 있다.

### 왜 필요한가?
테이블에 데이터가 많아질수록 조건에 맞는 값을 찾는 데 시간이 오래 걸릴 수 있다.
이때 인덱스가 있으면 모든 데이터를 처음부터 끝까지 읽지 않고, 더 빠르게 원하는 위치를 찾을 수 있다.

### 예시
```sql
-- user_missions 테이블에서
-- user_id와 status를 자주 조건으로 조회할 것 같아서 복합 인덱스 생성
CREATE INDEX idx_user_missions_user_status
ON user_missions (user_id, status);
```
특정 사용자의 진행중 미션, 완료 미션 같은 걸 더 빨리 찾기 위해 인덱스를 만든다.

- ```idx_user_missions_user_status```는 인덱스 이름
- ```(user_id, status)```는 이 두 컬럼을 기준으로 인덱스를 만든다는 뜻

### 인덱스를 걸어볼 수 있는 컬럼 예시

이번 미션 기준으로는 아래 컬럼들이 후보가 될 수 있다고 생각했다.
- stores.region_id
- missions.store_id
- user_missions.user_id
- user_missions.status
- reviews.user_id

### 인덱스의 장점
- 조회 속도가 빨라진다
- WHERE, JOIN, ORDER BY에 자주 쓰는 컬럼에서 효과가 크다
- 페이지네이션 성능에도 영향을 줄 수 있다

### 인덱스의 단점
- 인덱스를 저장할 공간이 추가로 필요하다
- INSERT, UPDATE, DELETE 시 인덱스도 같이 관리해야 해서 쓰기 성능이 느려질 수 있다
- 무조건 많이 건다고 좋은 것이 아니라, 자주 조회되는 컬럼에 맞게 걸어야 한다

### 느낀 점

인덱스는 “조회는 빠르게, 쓰기는 조금 느리게” 만드는 선택이라고 이해했다.
그래서 서비스에서 어떤 화면이 자주 조회되는지를 먼저 보고, 그 화면에서 자주 사용하는 컬럼을 기준으로 잡는 게 중요하다고 느꼈다.

## 4. ORM vs Raw SQL

### ORM이란

ORM은 Object Relational Mapping의 약자다.
객체와 테이블을 매핑해서 SQL을 직접 많이 작성하지 않아도 데이터를 다룰 수 있게 해주는 도구다.
Node.js에서는 Prisma, TypeORM, Sequelize 같은 ORM이 많이 언급된다.

### Raw SQL이란

Raw SQL은 SQL을 직접 작성해서 사용하는 방식이다.
```sql
-- users와 reviews를 연결해서
-- 특정 사용자가 작성한 리뷰 개수를 같이 조회하는 쿼리
SELECT
    u.id,                         -- 사용자 ID
    u.nickname,                   -- 사용자 닉네임
    COUNT(r.id) AS review_count   -- 리뷰 개수
FROM users u
LEFT JOIN reviews r
    ON u.id = r.user_id           -- users.id와 reviews.user_id를 연결
WHERE u.id = 1                    -- 1번 사용자만 조회
GROUP BY
    u.id,
    u.nickname;                   -- 사용자별로 묶어서 리뷰 개수 계산
```
1번 사용자의 닉네임과, 그 사용자가 작성한 리뷰 개수를 같이 보여주는 쿼리다.

```LEFT JOIN```

사용자 정보는 꼭 보여주고, 리뷰가 없어도 사용자는 나오게 하고 싶을 때 씀

- 리뷰가 있으면 개수가 계산됨
- 리뷰가 없어도 사용자 행은 남음

```ON u.id = r.user_id```
users 테이블의 id와 reviews 테이블의 user_id가 같은 것끼리 연결한다는 뜻

```GROUP BY```
리뷰가 여러 개면 여러 행이 생기기 때문에, 사용자 기준으로 다시 묶어서 개수를 세는 것

### ORM의 장점
- 생산성이 좋다
- 기본적인 CRUD를 빠르게 만들 수 있다
- 코드와 DB 구조를 어느 정도 일관되게 맞추기 쉽다
- 타입 지원이 좋은 ORM은 개발 편의성이 높다

### ORM의 단점
- 복잡한 쿼리에서는 답답할 수 있다
- 내부적으로 어떤 SQL이 나가는지 모르고 쓰면 성능 문제를 놓칠 수 있다
- SQL 감각이 약해질 수 있다

### Raw SQL의 장점
- 원하는 쿼리를 세밀하게 직접 제어할 수 있다
- 복잡한 조회, 집계, 튜닝에 유리하다
- 실제 DB가 어떻게 동작하는지 더 직접적으로 이해하게 된다

### Raw SQL의 단점
- 직접 작성해야 해서 번거롭다
- 쿼리가 길어지면 유지보수가 어려워질 수 있다
- 반복적인 CRUD는 비효율적일 수 있다

### 유명한 ORM
- Prisma

최근 Node.js에서 많이 쓰이는 ORM이다.
타입 안정성이 좋고 문서화가 잘 되어 있어서 입문자도 비교적 접근하기 편하다.

- TypeORM

엔티티 기반 ORM 느낌이 강하고, NestJS와 같이 언급되는 경우가 많다.
전통적인 ORM 스타일에 더 가깝다고 생각했다.

- Sequelize

Node.js에서 오래 사용된 ORM이다.
자료가 많고 사례가 풍부하지만, 최근에는 Prisma를 더 자주 보게 되는 것 같다.

### 느낀 점

ORM이 있으면 분명 빠르고 편하게 개발할 수 있다.
하지만 이번 주차를 하면서 느낀 건, ORM을 쓰더라도 결국 SQL을 이해하지 못하면 복잡한 조회나 성능 문제를 해결하기 어렵다는 점이었다.
그래서 지금 단계에서는 Raw SQL을 직접 짜보는 연습이 오히려 더 중요하다고 느꼈다.
