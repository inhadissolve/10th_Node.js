# 3주차 핵심 키워드 정리

## 1. API

API는 Application Programming Interface의 약자이다.  
처음에는 클라이언트와 서버가 통신할 때 쓰는 것 정도로만 생각했는데, 이번 주차를 하면서 API는 더 넓은 의미를 가진다는 걸 알게 되었다.

내가 이해한 API는 “복잡한 내부 구현을 감추고, 필요한 기능만 쉽게 사용할 수 있도록 꺼내놓은 인터페이스”이다.  
예를 들어 자바스크립트에서 `console.log()`를 쓸 때 우리는 내부적으로 콘솔 장치가 어떻게 동작하는지 알지 못해도 된다.  
그냥 함수를 호출하면 원하는 결과가 나온다. 이런 식으로 복잡한 과정을 숨기고 사용 가능한 형태로 제공하는 것이 API라고 이해했다.

즉, 백엔드에서 API를 만든다는 것은  
프론트엔드가 서버의 내부 구현을 몰라도 필요한 기능을 사용할 수 있게 약속된 형태로 제공하는 것이라고 생각했다.

---

## 2. REST API

REST는 Representational State Transfer의 약자이고, HTTP 메소드와 자원을 이용해 통신하는 웹 서비스 설계 방식이다.  
쉽게 말하면 URL과 HTTP 메소드를 활용해서 자원을 다루는 규칙이라고 이해했다.

예를 들어

- 사용자 생성은 POST
- 사용자 조회는 GET
- 사용자 수정은 PATCH 또는 PUT
- 사용자 삭제는 DELETE

처럼 HTTP 메소드와 역할을 연결해서 쓰는 방식이다.

REST API의 장점은 URL만 보아도 어느 정도 역할을 짐작할 수 있다는 점이다.  
프론트 입장에서도 API를 이해하기 쉽고, 서버 입장에서도 구조를 통일하기 좋다.

다만 이번 주차를 하면서 느낀 건, REST 규칙을 너무 딱딱하게 지키는 것보다 **프론트가 보고 이해하기 쉬운 설계가 더 중요하다**는 점이었다.  
예를 들어 로그인은 완전히 REST스럽게 표현하기 애매하기 때문에 `/users/login`처럼 설계해도 괜찮다는 점이 인상적이었다.

---

## 3. HTTP Method와 CRUD

이번 주차에서 다시 정리한 HTTP Method와 CRUD의 대응은 아래와 같다.

- Create → POST
- Read → GET
- Update → PUT, PATCH
- Delete → DELETE

여기서 PATCH와 PUT의 차이도 다시 보게 됐다.

### POST
새로운 자원을 생성하거나, 특정 처리를 요청할 때 사용한다.  
예를 들어 회원가입은 `POST /users` 또는 `POST /users/signup` 형태로 설계할 수 있다.

### GET
데이터를 조회할 때 사용한다.  
예를 들어 내 정보 조회, 미션 목록 조회 같은 기능이다.

### PUT
자원의 전체 수정에 가깝다.  
기존 정보를 전부 새 값으로 덮어쓰는 느낌이다.

### PATCH
자원의 일부 수정에 사용한다.  
예를 들어 닉네임만 바꾸거나, 회원 상태만 바꾸는 경우처럼 일부 필드만 수정할 때 적합하다.

### DELETE
자원을 삭제할 때 사용한다.  
다만 실제 서비스에서는 물리 삭제보다 소프트 삭제가 많기 때문에, 회원 탈퇴처럼 상태만 비활성화하는 기능은 DELETE보다 PATCH가 더 자연스러울 수 있다는 점을 알게 되었다.

내가 이해한 핵심은  
HTTP Method는 단순히 외워야 하는 문법이 아니라, **이 API가 어떤 동작을 하는지 첫눈에 보여주는 역할**을 한다는 점이었다.

## 4. RESTful URI 설계 규칙

워크북을 보면서 정리한 RESTful URI 설계 규칙은 아래와 같다.

### 1) URI에 동사를 넣지 않는다
예를 들어 `/getUsers`, `/createUser` 같은 식보다  
`/users`처럼 자원 자체를 표현하는 게 더 RESTful하다.

### 2) 자원은 보통 복수형으로 쓴다
사용자 관련 API면 `/users`,  
미션이면 `/missions`,  
리뷰면 `/reviews`처럼 복수형이 기본이다.

### 3) 특정 자원 하나를 가리킬 때는 식별자를 붙인다
예를 들어 특정 사용자라면 `/users/{userId}`  
특정 미션이라면 `/missions/{missionId}`

### 4) 자원 간 관계가 있으면 URI에 계층적으로 표현할 수 있다
예를 들어 특정 사용자의 미션 목록이면  
`/users/{userId}/missions`

### 5) 하지만 무조건 규칙만 따르기보다 의미 전달이 더 중요하다
예를 들어 로그인은 `/users/login`처럼 설계해도 괜찮다.  
REST 원칙만 고집해서 프론트가 헷갈리게 만드는 것보다, 목적이 명확하게 드러나는 설계가 더 낫다고 느꼈다.
