# 1주차 키워드 정리

이번 주차에서는 관계형 DB 설계에서 자주 나오는 개념들을 정리하고, 미션 ERD에 실제로 적용해봤다.

## 외래키
정의  
다른 테이블의 기본키를 참조하는 컬럼이다.

왜 필요한지  
테이블 간 관계를 DB 수준에서 보장하고, 잘못된 데이터가 들어오는 걸 막을 수 있다.

이번 미션 적용 예시  
store 테이블이 region의 id를 참조하도록 설계했다.

주의점  
삭제나 수정이 일어날 때 참조 무결성이 깨질 수 있어서 on delete 정책을 고려해야 한다.

## 기본키
정의  
각 row를 유일하게 식별하는 값이다.

왜 필요한지  
조회와 관계 설정의 기준이 된다.

이번 미션 적용 예시  
user, store, mission 등 모든 핵심 테이블에 id를 기본키로 두었다.

주의점  
자주 바뀌는 값이나 의미 있는 값을 기본키로 쓰면 운영 중 문제가 생길 수 있다.

## ER 다이어그램
정의  
테이블과 관계를 시각적으로 표현한 설계 결과물이다.

왜 필요한지  
팀 전체가 같은 DB 구조를 보고 개발할 수 있게 해준다.

이번 미션 적용 예시  
user와 mission은 다대다 관계라서 중간 테이블로 풀어서 ERD에 반영했다.

주의점  
처음부터 완벽할 필요는 없지만, 최소한 큰 틀과 관계는 초기에 맞춰야 한다.

## 복합 키
정의  
두 개 이상의 컬럼을 묶어 하나의 기본키로 사용하는 방식이다.

왜 필요한지  
중복을 막고 특정 관계의 유일성을 보장할 때 쓸 수 있다.

이번 미션 적용 예시  
user_mission 테이블에서 user_id와 mission_id 조합이 중복되지 않도록 유니크 제약 또는 복합키를 고려했다.

주의점  
조인이 많아지고 키가 길어질 수 있어서 대체키 id를 두고 유니크 제약을 거는 방법도 많이 쓴다.

## 연관관계
정의  
테이블 간 1대1, 1대다, 다대다 관계를 의미한다.

이번 미션 적용 예시  
region과 store는 1대다, user와 mission은 다대다로 설계했다.

## 정규화
정의  
중복을 줄이고 이상 현상을 막기 위해 테이블을 분리하는 설계 방식이다.

이번 미션 적용 예시  
mission 진행 상태와 완료 여부는 user_mission에 두고 mission 자체 데이터와 분리했다.

## 반정규화
정의  
성능이나 조회 편의 때문에 일부 중복을 허용하는 설계 방식이다.

이번 미션 적용 예시  
user 테이블에 현재 포인트를 컬럼으로 두는 방식을 고려했다.

주의점  
정합성 관리가 어려워지므로 업데이트 규칙과 검증이 필요하다.

## 느낀 점
이번 미션에서 다대다 관계를 중간 테이블로 풀어야 할 부분이 많아서 정규화의 필요성을 체감했다.
